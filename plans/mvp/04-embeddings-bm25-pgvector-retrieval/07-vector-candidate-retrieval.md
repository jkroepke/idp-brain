# 4.7: Vector Candidate Retrieval

## Goal
Implement pgvector candidate retrieval using configured query embeddings, filtered chunk scopes, exact vector ordering for selective subsets, and HNSW for broad dense retrieval.

## Prerequisites

- Step 4.1 is complete.

## Implementation Instructions
1. Implement `VectorCandidateRetriever.retrieve(query, filters, profile, limit)` using the embedding provider selected by the active query profile.
2. Generate a query embedding from the user query text and sanitized query metadata. Do not embed raw source chunks, raw file content, or ineligible context in the query embedding request.
3. Join `embeddings` to a filtered chunk scope before ordering by vector distance. The SQL shape must keep filters ahead of candidate exposure:
   ```sql
   SELECT embeddings.chunk_id,
          embeddings.embedding <=> :query_embedding AS distance
   FROM embeddings
   JOIN filtered_chunks ON filtered_chunks.id = embeddings.chunk_id
   WHERE embeddings.embedding_model_id = :embedding_model_id
     AND embeddings.index_version_id = :index_version_id
     AND embeddings.is_active = true
   ORDER BY embeddings.embedding <=> :query_embedding
   LIMIT :vector_top_k;
   ```
4. Use exact vector ordering for small or highly selective filtered subsets. Use HNSW-backed ordering for broad retrieval through the same SQL ordering and the indexes from Step 4.4.
5. Configure `hnsw.ef_search`, vector candidate count, embedding model ID, index version ID, and exact-search threshold through retrieval profile settings; do not hardcode those values in the retriever.
6. Return candidates with `retrieval_path = "vector"`, rank position, cosine distance, embedding model ID, index version ID, and sanitized metadata.
7. Do not convert vector distance into a fused score in this step. Reciprocal-rank fusion happens in Step 4.10.
8. Fail closed if the query profile references an external embedding provider and external model calls are not explicitly enabled.
9. Do not persist query embeddings unless a future retention policy explicitly allows it. Retrieval events may record provider ID, model ID, dimensions, and sanitized query hash only.
10. Add deterministic fixture vectors generated by the mock provider so CI can assert stable ranking without external APIs.

## Acceptance Criteria
- Vector retrieval uses pgvector over active sanitized chunk embeddings.
- Query embedding generation is deterministic in CI and external-provider gated in local or production use.
- Source allowlist, license, sensitivity, redaction, and version filters apply before vector candidates are exposed.
- Vector distances are not compared directly with BM25 scores.
- Query embeddings and source text are not persisted or logged.
